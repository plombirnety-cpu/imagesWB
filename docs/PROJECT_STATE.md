## Возврат гринскрина и контекстный chroma-предохранитель (2026-07-22, Codex)

**Жалоба владельца:** в живой партии Demon Slayer `f5465586dd54` все пять готовых
принтов вышли на синем фоне; Иноске с большой сине-серой массой волос/маски должен
был генерироваться на зелёном.

**Корень воспроизведён на реальных `design.json`:** LLM правильно писал в prose
`green chroma-key background`, но старый предохранитель в `art_director._parse`
искал голое слово `green` по всему prompt и принимал цвет ФОНА за зелёную часть
ПЕРСОНАЖА, после чего безусловно менял `chroma: green -> blue`. У Иноске дополнительно
есть маленькие зелёные глаза — ради них нельзя терять большие синие детали героя.

**Исправлено:** `_has_noticeable_green_subject_feature` сначала исключает управляющее
описание green chroma/background и малые детали (глаза, зрачки, украшения), затем
форсит blue только когда зелёный связан с заметной массой дизайна: волосы, кожа, мех,
одежда/хаори/броня, оружие, крупный эффект или типографика. На пяти живых паспортах
новая классификация: Тандзиро=`True` (синий верен из-за зелёного хаори), Тенген,
Зеницу, Незуко, Иноске=`False` (зелёный фон). Явный выбор `blue` сохраняется.

**Проверка и прод:** первоначальный harness краснел на green-background; после фикса
`CHROMA_GUARD_PASS`, аудит пяти живых prompt — `LIVE_CHROMA_AUDIT_PASS`, Python compile
зелёный, совместимый production-набор `test_chroma_selection + chroma_qc + style_bank`
— **63 passed**. Создан backup
`/opt/print-panel/backups/codex_20260722_chroma_guard/art_director.py`, образ пересобран,
контейнер `print-factory-panel` перезапущен и `healthy`, `/health` отвечает `ok`;
предыдущая партия в persistent volume сохранена. Исправление действует на НОВЫЕ
генерации; старые синие PNG автоматически не перекрашиваются.

## Исправление подмены франшизы и усиление стиля 34 (2026-07-22, Codex)

**Жалоба владельца:** в панели `http://195.133.66.37:8040` тема «Клинок
рассекающий демонов» выдала целую партию персонажей «Гачиакуты»; выбранный стиль

**Корень подмены воспроизведён детерминированно.** `franchise_scout._slugify`
оставлял только `[a-z0-9]`, поэтому ЛЮБОЕ полностью кириллическое название
становилось `untitled`. «Гачиакута» и «Клинок рассекающий демонов» читали один
дневной файл `untitled_<date>.json`; список Рудо/Энджина из первого запроса
переиспользовался вторым. Это объясняет именно целую неверную партию, а не
случайную ошибку Gemini.

**Исправлено локально:**

1. `franchise_scout._slugify` теперь нормализует название через NFKC+casefold.
   Чистые ASCII-тайтлы сохраняют прежние читаемые ключи, а Unicode-тайтлы получают
   стабильный 12-символьный SHA-256-суффикс (`title_<digest>` или
   `<ascii-part>_<digest>`). Старый общий `untitled` русскими темами больше не
   читается; аналогично изолирован кэш сырых сигналов.
2. Завершена незакоммиченная работа Claude по каноническим референсам:
   `franchise_scout` получает `title_ref` (romaji/English) из AniList;
   `panel/orchestrator.DesignTask` несёт `name_en + title_ref` из досье и после
   арт-директора принудительно исправляет его ненадёжные `character_en/title_en`;
   `character_ref` включает тайтл в ключ портретного кэша и выбирает AniList-кандидата
   по совпадению франшизы, а не только по общей популярности. Это закрывает случаи
   `Enjin -> engine` и одноимённых героев из разных аниме.
3. Стиль `34_anime_magazine_cover` приведён к шести утверждённым пробникам Slime:
   строгая вертикаль 2:3; крупный портрет по грудь (70–85% высоты); лицо резкое,
   руки преимущественно вне кадра; динамика через взгляд/волосы/компактные эффекты;
   фиксированные зоны катаканы/английских строк/вертикальных кандзи/печатей;
   каноническая палитра героя вместо грязного монохрома; идеально ровный green/blue
   chroma до краёв. Кроме банка стилей добавлен code-level quality hint, поэтому
   сетка попадает непосредственно в user-запрос арт-директору только для стиля 34.
4. Добавлены регрессионные тесты кэша, Unicode-нормализации, протяжки досье,
   title-aware AniList, раздельного кэша одноимённых персонажей и сетки стиля 34.

### Баг 1 — двойной текст в text-fallback ветке (`batch_print.py`, `render_design`)

**Точная механика бага (диагностирована по регресс-тесту тестировщика + докстрингу
живого кейса `0007_payback__тайская_дорама`, style_id `11_propaganda_B`)**: когда
ВСЕ основные попытки со встроенным текстом проваливают OCR (например временный сбой
самой OCR-модели), код уходил в text-fallback — ОДНА дополнительная генерация с
`design["type_spec"]=""` (`art_director.build_prompt` добавляет обычный безусловный
запрет букв `_NO_TEXT_TAIL`). НО `design["prompt"]` (основное художественное описание
сцены, часть JSON от Claude, НЕ трогается text-fallback веткой) для стилей типа
`11_propaganda_B`/`STYLE_BANK` УЖЕ подробно описывает встроенную типографику как
структурный элемент композиции ("Typography is integrated as a structural design
element: a single bold diagonal wordmark ... A vertical column of upright Japanese
katakana stands as a perfectly straight anchor"). nano-banana следует этому раннему
конкретному описанию сцены СИЛЬНЕЕ, чем позднему общему запрету `_NO_TEXT_TAIL`, и
рисует художественный текст ВСЁ РАВНО — fallback-картинка приходит с ВСТРОЕННЫМ
"PAYBACK" (диагональные плашки) + встроенной каной "ペイバック" (вертикальная колонка),
цветовой QC-гейт рамки (`_border_chroma_coverage`) проходит нормально (`coverage=
1.00`, гейт проверяет ТОЛЬКО цвет рамки кадра, не содержимое композиции). Код
БЕЗУСЛОВНО ставил `text_fallback=True` при входе в эту ветку и БЕЗУСЛОВНО применял
`apply_code_typography=True` (следствие `text_fallback=True`) — не проверяя, есть ли
УЖЕ текст на самой fallback-картинке. `typography_v3.compose_text_v3` (`text_modes_v3
=["kanji_on", "collection_footer"]`) рисовала ЕЩЁ ОДНУ вертикальную кодовую кану
"ペイバック" ПРЯМО ПОВЕРХ уже нарисованной диагональной надписи "PAYBACK" (буквы
"AYB" почти полностью закрыты) + отдельную мелкую кодовую подпись "PAYBACK" снизу —
итог нечитаем, два текстовых слоя конфликтуют. Задача правильно предположила ОБА
возможных механизма ("фолбэк тоже что-то нарисовала" И "не прошла проверку на
отсутствие текста") — реальность: механизм №1 подтверждён (модель игнорирует запрет),
а формальный механизм №2 ТОЖЕ был причиной, потому что такой проверки ("текста нет")
ВООБЩЕ НЕ СУЩЕСТВОВАЛО в коде — только цветовой гейт рамки, не про содержимое.

**Фикс (`batch_print.py`)**:
1. Новая `_verify_no_text(image) -> bool` — ОБРАТНАЯ `_verify_text`: транскрибирует
   картинку тем же `providers.verify_text_in_image` (дефолтный `_OCR_PROMPT`,
   доп. HTTP-вызов не нужен — OCR и так дешёвый текстовый, не image-вызов) и через
   новую `_transcript_has_no_significant_text(transcript)` решает, действительно ли
   на картинке нет значимого текста (явные отрицающие фразы модели типа "no text"/
   "no visible text" ИЛИ пустой/без единой буквы-цифры-CJK-глифа ответ после очистки
   пунктуации). Сбой самого OCR-вызова -> `False` (консервативно — не подтверждено,
   что текста нет).
2. Text-fallback ветка (`elif expected_phrases:`) переписана в ЦИКЛ до
   `_FALLBACK_NO_TEXT_MAX_ATTEMPTS=2` дополнительных генераций БЕЗ текст-блока.
   КАЖДАЯ fallback-попытка проверяется: (а) цветовой QC-гейт рамки — если хуже
   основной попытки, цикл СРАЗУ прерывается (безопасность из двенадцатого захода —
   не тратим лишний OCR-вызов/попытку на заведомо испорченный фон); (б) если цвет ОК —
   `_verify_no_text`. Лучшая по цвету попытка, ПОДТВЕРЖДЁННАЯ `_verify_no_text` как
   реально без текста, становится финальной раной, `text_fallback=True`, кодовая
   typography применяется как раньше (безопасно — картинка реально пустая).
3. Если НИ ОДНА из до 2 fallback-попыток не подтвердилась как "без текста" (модель
   упорно рисует текст несмотря на запрет, живой кейс 0007) — новый флаг результата
   `single_text_no_overlay=True`: raw принимается КАК ЕСТЬ (лучшая по цвету рамки
   fallback-попытка, либо основная попытка, если фолбэк оказался хуже по цвету),
   `text_fallback` остаётся `True` (фолбэк-путь реально был пройден — факт хода
   пайплайна), но `apply_code_typography` теперь `not single_text_no_overlay and
   (...)` — БЕЗУСЛОВНО `False` при `single_text_no_overlay=True`, ДАЖЕ хотя
   `text_fallback=True` обычно включил бы кодовую типографику. Итог: принт выходит
   с ОДНИМ уже нарисованным художественным текстовым слоем, БЕЗ второго кодового —
   "лучше без повторного текста, чем двойной" (дословно по формулировке задачи; текст

**Корневая причина**: `typography_v3._ring_radius_and_center` считал радиус кольца как
половину диагонали bbox АЛЬФА-КАНАЛА ВСЕЙ картинки (`typography._alpha_bbox`) — bbox
включал любые визуальные элементы, торчащие ЗА пределы декоративного кольца (аура/
молнии — штатная часть `STYLE_BANK.json "09_ring_medallion".essence`, "crackling neon
aura energy pulses... streaking upward"). Диагональ раздувалась молниями, расчётный
радиус оказывался в ~2.5 раза больше фактического радиуса нарисованного кольца.

**Фикс (`typography_v3.py`)**:
1. Новая `_detect_ring_radius(rgba, cx_local, cy_local, fallback_radius)` — определяет
   ФАКТИЧЕСКИЙ радиус нарисованного кольца по радиальному угловому профилю альфа-канала:
   для каждого радиального бина (2px) считает долю занятых угловых секторов (180 бинов
   по 2°), а не площадь/долю пикселей. Декоративное кольцо — почти замкнутая окружность,
   даёт покрытие близкое к 100% на своём радиусе; аура/молнии (несколько лучей в разных
   направлениях) даже физически покрывая больше площади, на ЛЮБОМ отдельном радиусе дают
   низкое угловое покрытие. Порог `_COVERAGE_THRESHOLD=0.55` откалиброван на синтетике
   регресс-теста (8 лучей аур дают заметно ниже порога на любом радиусе).
2. Различает "реальное отдельное кольцо" (последняя непрерывная группа "кольцевых"
   бинов, отделённая ВИДИМЫМ РАЗРЫВОМ от основной массы фигуры) от "сплошной заливки"
   (обычный круглый силуэт БЕЗ отдельного кольца, группа тянется от центра фигуры без
   разрыва) — для второго случая используется СТАРАЯ формула (`fallback_radius`,
   половина диагонали bbox * 1.08), чтобы не сломать дизайны/тесты без style_id=
   09_ring_medallion. Для найденного разрыва — внешний край группы * 1.10 (небольшой
   запас, буквы чуть снаружи кольца, не впритык на крайний пиксель).
3. `_ring_radius_and_center` теперь принимает саму `rgba` (не только bbox-координаты) и
   вызывает `_detect_ring_radius`; единственный вызов внутри `ring_text` обновлён.
4. Новый импорт `numpy as np` в `typography_v3.py` (уже в requirements.txt, используется
   в `palette.py` — не новая зависимость проекта).

**Тесты**: `tests/test_ring_text_regression_qa.py` (2 теста тестировщика) — оба зелёные

**Ключевая находка (функциональный пробел, не просто "модель иногда ошибается"):**
`batch_print._verify_text` проверял ТОЛЬКО substring-вхождение ожидаемой фразы в OCR-
транскрипте — у него не было механизма отклонить попытку при ДУБЛИРОВАНИИ фразы.
Дизайн 02 прошёл QC-цикл (`border coverage=1.00`, OCR "OK") с первой попытки,
`text_fallback=False`, никакого ретрая не случилось — реальный OCR-транскрипт
(`providers.verify_text_in_image` вызван напрямую тестировщиком) буквально содержал
фразу дважды: `"I WILL NEVER STOP\nNO MATTER WHAT\nI will never stop\nno matter what"`.
QC-цикл в принципе не мог поймать эту регрессию, даже если бы дубль появился на всех
попытках подряд.

**Фиксы:**

1. **`batch_print.py`** — новая `_count_nonoverlapping(haystack, needle)` (обёртка над
   `str.count`, непересекающиеся вхождения). `_verify_text` теперь считает вхождения
   КАЖДОЙ ожидаемой фразы в нормализованном транскрипте — БОЛЬШЕ ОДНОГО вхождения =
   провал (`duplicated`), наравне с отсутствием фразы (`missing`); лог явно называет
   причину (`"фразы повторены ДВАЖДЫ+ (дубль на картинке)"`). Работает одинаково для
   `slogan`/`quote`/`name_jp` (не выделяет кандзи-колонку как особый случай — дубль
   кандзи тоже провалит проверку). Проваленная попытка обрабатывается ТЕМ ЖЕ ретрай/
   фолбэк механизмом `render_design`, что и раньше — никаких изменений в самом цикле
   не понадобилось, весь фикс инкапсулирован внутри `_verify_text`.
2. **`art_director.py`** (`_text_render_block`) — ТРЕТЬЕ усиление инструкции (после
   одиннадцатого захода, которое НЕ предотвратило регресс на живой генерации): явно
   называет паттерн "маленькое эхо/тизер фразы рядом с фигурой + отдельная крупная
   копия внизу" как anti-pattern по имени ("do NOT add a small echo or teaser copy of
   the phrase near the figure in addition to the main lettering placement — there is
   only ONE lettering placement for this phrase on the entire composition") + закрывает
   лазейку перефразирования той же фразы другими словами.
3. **`art_director.py`** (`_TYPE_SPEC_SCHEMA`) — новое явное правило: `type_spec`
   ДОЛЖЕН описывать РОВНО ОДНУ зону размещения для главной фразы, не две разных места
   (первопричина живого бага — Claude сам написал в `type_spec` ДВЕ зоны для одной
   quote-фразы: `"quote text ... tucked tightly below the figure's feet"` ПЛЮС отдельно
   `"placed tilted along the bottom of the composition"` — модель честно нарисовала
   обе). Кандзи-колонка (`name_jp`) явно исключена из этого правила — она отдельный
   блок, не вторая зона самой фразы.
4. **`art_director.py`** (`_chroma_bg`) — новое явное предложение: запрет белой/светлой
   sticker-style обводки, "hugging the silhouette" персонажа или эффектов ауры/пламени

   этот дефект по конструкции (проверяет только крайнюю кромку кадра) — это ожидаемо

GENJUTSU" вместо полной фразы. Фикс: текст растёт НАРУЖУ от внешнего края катаканы

кропом). Второй попутный баг в этом же месте: `sub_min_size` изначально совпадал с

не могла реально уменьшить кегль. Заведён отдельный `sub_min_size=2%`. Оба фикса

## Шестой заход (2026-07-08) — рисование ПО РЕФЕРЕНСУ (character_ref.py)

**Что сделано (5 файлов, `theme_scout.py`/`franchise_scout.py` НЕ трогались — они

1. **`character_ref.py`** (новый файл) — `get_reference(character_en, title_en="")

## Пятый заход (2026-07-08) — фикс: graceful degradation при сбое Claude API

Тестировщик поймал реальный баг кода (не внешний блокер): при исчерпанном балансе
Anthropic (`credit balance too low`, HTTP 400) весь `daily_prints.py` падал
необработанным `anthropic.BadRequestError` traceback вместо заявленной graceful
degradation — потому что `theme_scout._ask_claude_scout` и
`franchise_scout._ask_claude_dossier` оборачивали в try/except только ПАРСИНГ
JSON-ответа, но НЕ сам сетевой вызов `client.messages.create()`. Любой сбой Claude
API (не только баланс — сеть, rate-limit, 5xx) ронял весь дневной контур.

**Фикс (2 файла, точечно вокруг `client.messages.create`):**
- `theme_scout.py::_ask_claude_scout` — `client.messages.create` обёрнут в
  `try/except anthropic.APIError`, при сбое печатает предупреждение и возвращает
  `""` (тот же сигнал, что и невалидный JSON-ответ — `_parse_scout("")` уже
  возвращает `None`, не `[]`, проверено). Дальше срабатывает СУЩЕСТВУЮЩИЙ путь:
  `_ask_and_parse_with_retry` ретраит один раз, при двойном сбое
  `expand_trends_to_tasks` печатает предупреждение и пропускает ТОЛЬКО этот блок
  (media/anime либо конкретную pop-группу), план строится из остального + evergreen.
- `franchise_scout.py::_ask_claude_dossier` — та же обёртка, тот же контракт
  (`_parse_dossier("")` тоже даёт `None`). При двойном сбое срабатывает
  СУЩЕСТВУЮЩАЯ явная `RuntimeError` в `_ask_and_parse_dossier_with_retry` (не
  тихий откат, дамп сырого ответа в файл — здесь дампится текст ошибки API, что
  само по себе диагностично). `_collect_dossiers` в `theme_scout.py` эту
  `RuntimeError` уже ловил и раньше (не трогалось) — падение ОДНОГО тайтла
  по-прежнему не мешает остальным.
- Ничего не глушится тихо: обе ветки логируют предупреждение через `print`, ловят
  именно `anthropic.APIError` (базовый класс всех сбоев API/сети/таймаутов SDK), не
  голый `Exception` — программные баги внутри блока (не связанные с самим сетевым
  вызовом) по-прежнему всплывают.

**Регресс-тесты (4 новых, `tests/test_theme_scout.py` +2, `tests/test_franchise_scout.py`
+2, все офлайн — мокают `anthropic.Anthropic` на клиент, кидающий
`anthropic.APIConnectionError`):**
- `_ask_claude_scout` при сбое API возвращает `""`, не пробрасывает исключение.
- `_ask_and_parse_with_retry` сквозь настоящий `_ask_claude_scout` (не подменённый)
  переживает ДВОЙНОЙ сбой API — 2 попытки (проверено счётчиком), результат `None`/`[]`,
  без исключения наружу.
- `_ask_claude_dossier` при сбое API возвращает `""`.
- `build_dossier` при ДВОЙНОМ сбое API даёт ту же `RuntimeError` с тем же
  сообщением, что и при сбое парсинга (не сырой `anthropic.APIConnectionError`).

**Прогнано реально:**
1. `python -m pytest tests/ -v` — **27/27 зелёных**, 2.4с, полностью офлайн.
2. `python franchise_scout.py "Re:Zero kara Hajimeru Isekai Seikatsu"` (реальная
   сеть, реальный баланс Anthropic по-прежнему исчерпан) — вместо traceback:
   2 аккуратных предупреждения (по одной на попытку) + ожидаемая `RuntimeError` +
   `exit code=1` (корректно для CLI — баланс реально не восстановлен, ошибка
   осталась явной, просто теперь через контролируемый путь модуля).
3. `python daily_prints.py --dry-run --limit 15` (реальная сеть, тот же
   исчерпанный баланс) — **EXITCODE=0** (было 1 до фикса). Все 3 франшизы дня
   (Mushoku Tensei S3, One Piece, третий тайтл) + общий блок media/anime + ВСЕ
   pop-группы (anilist/jikan/youtube/gtrends/tmdb/applemusic/mdl) корректно
   поймали двойной сбой Claude API, продолжили работу, план деградировал до
   чистого evergreen-добора (312 → 15 по `--limit`), смета/журнал/dry-run

4. Тестовые артефакты сегодняшних прогонов (`out_batch/daily_2026-07-08/`,

**Важно для приёмки — что НЕ входило в эту задачу и по-прежнему блокировано:**
Баланс Anthropic API (`ANTHROPIC_API_KEY`) по-прежнему исчерпан — это внешний

исчерпания баланса, см. запись ниже) — сейчас его не перепроверить без пополнения
баланса, но фикс этого раунда не про ранжирование, а про то, что отказ API теперь
НЕ роняет весь дневной контур необработанным traceback. После пополнения баланса
рекомендуется владельцу/лиду перезапустить полный чек-лист (шаги 2-6 из отчёта
тестировщика) ещё раз, чтобы увидеть содержательные досье, а не только graceful

## Известные ограничения / блокеры (актуально на конец второго захода)

**ГЛАВНЫЙ БЛОКЕР: генерация картинок не работает ни на одном провайдере на момент
сдачи.**
- `pollinations`: `POLLINATIONS_TOKEN` (общий на все проекты студии) — HTTP 402,
  баланс `0.0000` на модели `nanobanana` (перепроверено повторно в этом заходе, тот же
  результат, что в первом заходе).
- `gemini`: ключ не заведён для этого проекта (см. блок 1 выше — блокировка
  permissions на переносе чужих ключей без личного разрешения владельца).

**Как разблокировать (нужно одно из):**
1. Владелец лично подтверждает (в сообщении, не через координатора) разрешение
   использовать один из существующих Gemini-ключей студии для print-factory-nb —
   тогда следующий заход впишет ключ, протестирует, повторит smoke.
2. Владелец выдаёт новый выделенный `GEMINI_API_KEY`.
3. Владелец пополняет баланс `POLLINATIONS_TOKEN`.

**После разблокировки — что нужно доделать (не блокеры кода, просто следующие шаги):**
1. Smoke на «Кенпачи Зараки (Блич)», `--format diecut` — единственная непройденная
   базовая проверка с первого захода.
2. Блок 2: `python batch_print.py --file themes_test.txt --format diecut` (все 4
   темы), путь к результату — координатору для визуального ревью.
3. Блок 3, боевой мини-тест: `python daily_prints.py --limit 2` (2 реальные платные
   генерации через полный дневной контур).

**Второстепенное (не блокер):**
- Тематизатор `theme_scout.py` вероятностно (не 100% стабильно) не укладывается в
  JSON за 1 вызов на очень насыщенных лентах трендов — контур устойчив (evergreen-
  fallback), но доля тем "из трендов" в плане дня может колебаться от прогона к
  прогону. Если для реального 500/день нужна БОЛЬШАЯ доля именно трендовых тем (не
  evergreen) — можно снизить `_MAX_TASKS_PER_CALL` ниже 60 или сделать несколько
  вызовов тематизатора подряд с разными подмножествами трендов (не реализовано в
  этом заходе — не было явного требования в ТЗ добиваться конкретной доли).
- `evergreen_themes.txt` — 104 строки, при `PRINTS_PER_DAY=500` и полном отсутствии
  трендовых тем round-robin повторит эти 104 темы ~3 раза за один день (защита от
  бесконечного цикла — максимум 3 полных прохода по списку, поэтому реальный потолок
  добора с пустыми трендами — 312, не 500; если нужно строго 500 при пустых трендах —
  увеличить множитель в `theme_scout.build_daily_plan` или добавить строк в файл).
