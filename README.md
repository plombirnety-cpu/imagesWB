# print-factory-nb

Конвейер генерации принтов для футболок через платный облачный API **nano-banana**
(Google Gemini Image). Зеркальный аналог self-hosted `comfyui-print-server/`, но без
своего GPU — картинку рисует облако. Два режима использования:

1. **Разовый батч** (`batch_print.py`) — конкретный список тем в файле, запуск руками.
2. **Ежедневный контур** (`theme_scout.py` + `daily_prints.py`) — тренды дня
   автоматически разворачиваются в план до 500 принтов/день, с бюджетным
   предохранителем и журналом прогресса (можно прервать/продолжить, не платя дважды).

## Пайплайн одного дизайна

```
тема -> Claude-арт-директор (идея + промпт + слоган + kana-имя + character_en/title_en
        + signature_props + text_mode)
     -> (если character_en распознан) character_ref.py: каноничный референс-портрет
     -> nano-banana рисует дизайн на хромакей-фоне (по тексту, либо ПО РЕФЕРЕНСУ)
     -> вырезка фона кодом (chroma_remove.py)
     -> слоган накладывается КОДОМ (typography.compose_text — типографика v2)
     -> 4 файла на дизайн
```

На каждый дизайн:
- `NN_raw.png` — как сгенерилось (сырой хромакей-фон);
- `NN_diecut.png` — прозрачный PNG (вырезка + типографика + juice-цвет);
- `NN_ongreen.png` — та же вырезка на ровном зелёном фоне RGB(0,177,64);
- `NN_design.json` — дамп идеи/промпта/слогана/kana/chroma/character_en/title_en/
  signature_props/text_mode для воспроизводимости.

Ядро пайплайна одного дизайна — функция `batch_print.render_design()`, переиспользуется
и разовым батчем, и ежедневным оркестратором (не продублировано).

## Каноничное оружие/атрибут персонажа (`signature_props`)

**Проблема**: арт-директор описывал оружие персонажа обобщённо («a sword»), из-за чего
на принте выходило не то оружие, что в каноне (пример: у Кенпачи Зараки вместо длинного
зазубренного нодати — обычный широкий тесак).

**Решение**: арт-директор (`art_director.py`) возвращает ещё одно поле —
`signature_props`: короткое английское описание канон-оружия/ключевого атрибута
персонажа с 2-3 опознавательными признаками формы/цвета/отделки (например для
Кенпачи Зараки: `"his nodachi: an extremely long, thin, heavily chipped and jagged
blade with a rectangular guard wrapped in bandages"`). Пустая строка, если у
персонажа/темы нет знакового предмета. Систем-промпт арт-директора требует описывать
оружие С ТОЙ ЖЕ строгостью, что и внешность персонажа — не обобщать до «a sword».

Если `signature_props` непусто, `art_director.build_prompt()` вставляет явное
предложение: *«The character's signature weapon/prop must match canon exactly:
\<signature_props\>.»* — сразу после художественного промпта, перед хромакей-хвостом.
При рисовании ПО РЕФЕРЕНСУ (`character_ref.py`) `_REFERENCE_PREFIX`
(`batch_print.py`) тоже требует точное соответствие канону: *«...and the character's
signature weapon exactly as in canon.»*.

## Форматы (`--format`)

- `cutout` — просто персонаж на хромакее, без обрамления и без художественного слогана.
- `diecut` — персонаж в обрамлении пламени/энергии, образующем внешний силуэт дизайна,
  + слоган-катчфраза снизу (стиль см. ниже).

## Референсы персонажей (`character_ref.py`)

**Проблема**: аниме-персонажи по одному текстовому промпту выходят «по мотивам» — не
похожими на канон (например у Кенпачи Зараки нет фирменной повязки на глазу), хотя
реальные объекты (машины и т.п.) nano-banana рисует отлично без референса. Причина:
gemini-2.5-flash-image принимает на вход не только текст, но и изображение — и умеет
перерисовать персонажа в новой позе, СОХРАНИВ опознаваемость лица/причёски/костюма,
если дать ей узнаваемый портрет-эталон.

**Решение** — рисование ПО РЕФЕРЕНСУ вместо чистого текста:

1. Арт-директор (`art_director.py`) теперь возвращает для каждого дизайна ещё два
   поля: `character_en` (имя персонажа латиницей, напр. `"Kenpachi Zaraki"`; пустая
   строка, если тема не про конкретного вымышленного персонажа) и `title_en`
   (франшиза латиницей, напр. `"Bleach"`).
2. Если `character_en` непусто, `batch_print.render_design()` вызывает
   `character_ref.get_reference(character_en, title_en)` — функция ищет каноничный
   портрет персонажа:
   - Источник 1 — **Jikan** (MyAnimeList): `GET /v4/characters?q=<character_en>&limit=10`,
     выбор лучшего совпадения по точности имени + максимуму `favorites`; при
     неоднозначности топ-1/топ-2 (близкие favorites) — доп. запрос
     `GET /v4/characters/{id}/anime`, побеждает кандидат, у которого встречается
     `title_en`.
   - Источник 2 (fallback) — **AniList** GraphQL `Character(search: ...) { image { large } }`.
   - Кэш на диске: `data/char_refs/<slug>.jpg` — повторный запрос читает файл, сеть не
     трогает.
3. При успехе — к промпту СПЕРЕДИ добавляется явный блок: используй референс как
   ТОЧНУЮ личность персонажа (лицо/причёска/костюм), но нарисуй НОВУЮ позу/композицию,
   не копируй позу/фон референса. Референс кодируется в base64 JPEG (сжат до 768px по
   большей стороне) и уходит в Gemini-запрос ПЕРЕД текстом промпта
   (`providers.generate_image(..., reference=...)`, только провайдер `gemini` —
   `pollinations` референс игнорирует с предупреждением, генерирует как раньше).
4. Любой сбой (сеть недоступна, персонаж не найден ни на Jikan, ни на AniList) —
   `get_reference` возвращает `None` с предупреждением в консоль, генерация идёт по
   чистому тексту, как раньше (graceful degradation, ничего не падает).

QC-гейт границ кадра, ретраи, вырезка фона и типографика слогана работают одинаково
и с референсом, и без — референс влияет только на сам вызов генерации картинки.

## Текст в генерации (`.env: TEXT_RENDER`, десятый заход)

**Проблема с кодовой типографикой**: до этого захода буквы ВСЕГДА рисовались кодом
(`typography.py`/`typography_v3.py`) ПОСЛЕ генерации, потому что ранние поколения
nano-banana (`gemini-2.5-flash-image`) рисовали текст с орфографическими ошибками
или вообще не рисовали (см. грабля №3 ниже — теперь пересмотрена). A/B оркестратора
(2026-07-08, `out_batch/ab_models/`, три модели на одном эталонном промпте с текстом)
подтвердил: новые поколения (`gemini-3.1-flash-image` = Nano Banana 2) рисуют
встроенный текст БЕЗ ошибок спеллинга и гармоничнее кодовой типографики — буквы
органично ложатся на форму дизайна, а не приклеены поверх.

**Решение — `.env: TEXT_RENDER` (дефолт `image`)**:

| `TEXT_RENDER` | Что делает |
|---|---|
| `image` (ДЕФОЛТ) | Текст ВСТРАИВАЕТСЯ прямо в генерацию nano-banana. Систем-промпт арт-директора (`art_director.py`) вместо безусловного запрета букв требует ВСТРОЕННУЮ типографику через новое поле схемы `type_spec` (стиль леттеринга по `mood`/размещение относительно фигуры/цвета словами из палитры сцены — см. таблицу ниже), `build_prompt` добавляет EXACT-SPELLING инструкцию («Spell the phrase EXACTLY, letter by letter: \<фраза\>») + «No other text anywhere». После генерации `batch_print._verify_text` — OCR-контроль ДЕШЁВОЙ текстовой моделью `gemini-2.5-flash` (`providers.verify_text_in_image`) сверяет транскрипт с ожидаемой фразой; при провале — ретрай генерации (общий счётчик `attempts` с QC-гейтом границ), при исчерпании — ЧЕСТНЫЙ ОТКАТ: одна доп. генерация БЕЗ текст-блока + типографика КОДОМ (`typography_v3`/`typography`), как раньше. |
| `code` | Старое поведение 1:1 (девятый заход и раньше) — текст ВСЕГДА запрещён в артворк (`no letters, no words, no typography...`), накладывается кодом `typography.py`/`typography_v3.py` после генерации. Остаётся рабочим фолбэком. |

**Поле `type_spec`** (новое в JSON-схеме `art_director.py`, применяется только при
`TEXT_RENDER=image`) — англ. описание стиля/размещения/цветов встроенной
типографики, БЕЗ самих слов слогана (сама фраза берётся из `quote`/`slogan`):

| `mood` | Стиль леттеринга в `type_spec` |
|---|---|
| `duotone_quote` | bold aggressive brush-graffiti lettering |
| `fashion_editorial` | elegant high-contrast serif display lettering (Playfair-style) |
| `pop_trash` | gothic blackletter-style lettering |
| без `mood` / простые темы | крепкий street capital леттеринг |

Пустой `type_spec` — эквивалент `text_mode='none'` и `text_modes_v3=[]`: дизайну
текст не нужен, `build_prompt` вставляет старый безусловный запрет букв (правило
«иногда текст не нужен» сохраняется буквально — арт-директор реально выбирает этот
вариант в ~20-30% дизайнов, как и раньше).

**Выбор exact-spelling фразы** (`art_director._exact_spelling_phrase`): приоритет
`quote` (типографика v3, цитата обычно длиннее/содержательнее) над `slogan` (v2).
Если есть `name_jp`/`kana` — текст-блок дополнительно просит вертикальную колонку
японской каллиграфии рядом с фигурой (тот же приём, что эталонный A/B-промпт
оркестратора).

**Защита букв при вырезке** (`batch_print.drop_small_islands`, параметр
`protect_text_islands`): встроенный текст у нижнего/бокового края диеката часто
рисуется как ОТДЕЛЬНЫЕ альфа-острова (буквы/апостроф не касаются основного силуэта
персонажа) — старый порог `min_frac` считался ОТ площади главного силуэта, из-за
чего целые буквы могли попасть под удаление наравне с мусорными водяными знаками.
При `protect_text_islands=True` (включается автоматически в `render_design`, когда
`TEXT_RENDER=image` и текст реально ожидается и НЕ ушёл в code-фолбэк) остров НЕ
удаляется, если его площадь `>= 0.04%` площади всего кадра — совсем крошечный
шумовой остров (1-2px) по-прежнему срезается.

**Когда кодовая типографика всё-таки применяется при `TEXT_RENDER=image`**:
1. `text_fallback=True` — OCR-контроль не сошёлся за все попытки (см. таблицу выше).
2. У дизайна вообще нет `type_spec` (тексту не место) — `typography_v3`/`typography`
   в этом случае тоже ничего не рисуют (`text_mode='none'`/`text_modes_v3=[]`), но
   технически путь остаётся «кодовым» на случай воспроизведения старых дампов через
   `reprocess_typo.py` (design.json без `type_spec` — обратная совместимость).

Если текст встроен генерацией И подтверждён OCR — код НЕ накладывает НИЧЕГО поверх
вырезки, диекат = вырезка «как есть» (`chroma_remove.cutout_green` +
`drop_small_islands` + `juice`).

**Модель** (`.env: GEMINI_MODEL`, дефолт `gemini-3.1-flash-image` = Nano Banana 2) —
переключено вместе с `TEXT_RENDER` по итогам A/B: заметно лучше и текст, и общая
гармоничность композиции при той же цене, что `gemini-2.5-flash-image`.
`GEMINI_MODEL_PREMIUM` (дефолт `gemini-3-pro-image`, Nano Banana Pro) — задел на
будущее, `providers.generate_image`/`_generate_gemini` принимают параметр `model`
для точечного переключения на премиум, использование НЕ обязательно в этом заходе
(боевой путь идёт через `GEMINI_MODEL`). `COST_PER_IMAGE_USD["gemini"]` оставлена
`$0.04` без изменений — уточнить по факту биллинга NB2, официальный прайсинг под
рукой не было на момент переключения.

## Типографика v2 — текст КАК ЧАСТЬ КОМПОЗИЦИИ (`typography.compose_text`)

**Проблема**: типографика v1 (5 стилей — см. раздел ниже) ужимала слоган в отдельную
строку в ПУСТОЙ полосе снизу дизайна — визуально оторвано от самой картинки, «убого»
по оценке владельца при повторной приёмке.

**Решение** — новая функция `typography.compose_text(figure_rgba, text_mode, slogan,
color_key, kana)`, которую арт-директор (`art_director.py`) выбирает под каждый
конкретный дизайн полем `text_mode`:

| `text_mode` | Что делает |
|---|---|
| `none` | Тексту не место — дизайн самодостаточен. Арт-директор реально выбирает этот режим примерно в 20-30% дизайнов (не формально), если композиция и так цельная. |
| `under` | Стритвир-плакат: 1-2 ударных слова слогана ОГРОМНЫМ кеглем (Anton/Archivo Black) размещаются ЗА фигурой — персонаж частично перекрывает буквы (текстовый слой рисуется ПЕРВЫМ, фигура наклеивается поверх). Ширина текста ~= ширине фигуры, верхняя треть композиции. |
| `punch` | Короткий слоган 1-3 строки с балансной разбивкой ПО СМЫСЛУ (ключевое слово крупнее остальных в 1.5-2x), наклон -6..-10°, размещение ВПЛОТНУЮ к нижней части фигуры с лёгким перекрытием силуэта (текст рисуется ПОВЕРХ фигуры). |
| `kana_side` | Вертикальная катакана крупно вдоль левого/правого края фигуры (сторона выбирается детерминированно по длине kana), ПРИЖАТА к фигуре с частичным перекрытием (~35% ширины колонки), + маленький латинский слоган под ней, растущий НАРУЖУ от силуэта (не под него, иначе обрезается широким низом фигуры). Без `kana` деградирует в `punch`. |

Ключевая геометрия: позиционирование текста — ОТ BOUNDING BOX'А АЛЬФЫ ФИГУРЫ (реальные
края персонажа на конкретной вырезке), а не от процентов холста, как в v1. Холст
расширяется ВНУТРИ `compose_text` по необходимости (замена блоку `если высота < 1.15
ширины -> extended` в `batch_print.py`, который использовался только для v1). Итог
кропится до реального контента (текст+фигура) с полями 4-6% — без мёртвой пустой
полосы. Минимальный кегль ударного текста (`under`/`punch`) — не меньше 7% ширины
фигуры (`_MIN_FONT_FRAC`); если строка не влезает — код переразбивает по словам
(меньше слов на строку / больше строк), а не ужимает кегль тише порога.

Слоганы стали короче ПО ОРИЕНТИРУ: 1-4 слова ударной сути (Кенпачи -> `LETS PARTY`);
длинная каноничная фраза (`COME ON LETS PARTY`) допустима, только если реально нужна
для `text_mode=punch` с балансной разбивкой на 2-3 строки.

**CLI/API — обратная совместимость**: `--text-style` по умолчанию `auto` — типографика
v2, режим берётся из `design["text_mode"]`. Явный стиль из v1 (`none`/`anton`/`kana`/
`comic`/`tag`, старые `STYLES`/`apply_style` в `typography.py` НЕ удалены и не
изменены) ПРИОРИТЕТНЕЕ `text_mode`, если передан явно — старые вызовы/интеграции
продолжают работать как раньше.

**Бесплатная проверка на уже оплаченных картинках** — `reprocess_typo.py`:
```powershell
python reprocess_typo.py --src out_batch/20260708_153106 --dst out_batch/typo_v2_demo
python reprocess_typo.py --src out_batch/20260708_153106 --dst out_batch/typo_v2_demo --mode punch
```
Читает `NN_raw.png` + `NN_design.json` из `--src` (без повторного платного вызова
Claude/nano-banana), прогоняет тот же пайплайн вырезки, что `batch_print.render_design`
(`chroma_remove.cutout_green` + `drop_small_islands` + `juice`), и НОВУЮ типографику.
Если в `design.json` есть `text_mode` — используется он; для СТАРЫХ дампов без этого
поля (до седьмого захода) режимы назначаются по кругу (`under`/`punch`/`kana_side`),
чтобы показать все три режима на одном батче. `--mode <режим>` форсирует один режим на
все дизайны батча.

## Типографика v3 — текст ИЗ ЦВЕТОВ ИЛЛЮСТРАЦИИ (`typography_v3.compose_text_v3`)

**Проблема с v2** (см. `docs/PRINT_STYLE_GUIDE.md` раздел 0 за полным разбором):
цвет текста в v2 брался из фиксированной палитры `_TITLE_COLORS` (6 хардкод-цветов
red/orange/white/yellow/purple/black), не из самой иллюстрации — партия принтов
выглядела "одним и тем же шрифтом с одной покраской". Одна шрифтовая роль на весь
принт. Нет структурного подвала-этикетки и нет режима "цитата в кавычках".

**Решение** — новый модуль `typography_v3.py` (не переписывает `typography.py`,
импортирует его низкоуровневые примитивы) + новый модуль `palette.py`
(`extract_palette` — свой Lloyd's k-means на numpy, без sklearn, детерминированный):

1. **Палитра из иллюстрации** (`palette.extract_palette(rgba, n=4)`) — k-means по
   пикселям с `alpha>20`, отсортирован по занимаемой площади, роли `dominant`/
   `accent`/`light`/`dark` назначаются программно (`palette.PaletteRoles`). Заливка
   текста и обводка/тень — ВСЕГДА из этой палитры, никогда `typography._TITLE_COLORS`.
2. **Шрифтовые роли v3** (2-3 на принт, файлы в `fonts/`, см. `fonts/FONTS.md`):
   `quote` (Permanent Marker), `quote_alt` (Caveat Brush), `display` (Playfair Display,
   variable-шрифт, вес 900 фиксируется кодом), `gothic` (Cinzel, variable, вес 900),
   `gothic_heavy` (UnifrakturCook, запасной вариант, не используется в мелком подвале).
3. **Композиционные режимы** (`typography_v3.TEXT_MODES_V3`), могут комбинироваться
   (кроме `editorial`):

| Режим | Что делает |
|---|---|
| `quote_bottom` | Цитата в кавычках под фигурой, слова ЧЕРЕДУЮТСЯ `accent`/`dominant` (начиная с `accent`), кавычки — цвет первого/последнего слова, обводка единая `dark`/`light` по контрасту с локальной яркостью фона. |
| `kanji_on` | Вертикальная колонка кандзи/катаканы имени НА фигуре (перекрытие почти полное, в отличие от v2 `kana_side` сбоку) — заливка `light` с плотной обводкой `dark`; вариант `ghost` (mood=`pop_trash`) — полупрозрачная заливка `accent`/`light` без обводки. |
| `editorial` | Display-заголовок (`character_en`), переплетённый с фигурой по глубине (упрощённая v3.0-реализация: полный проход ПОД фигурой + дублирование хвоста слова ПОВЕРХ) + подзаголовок катаканой/кандзи + мини-лого "Aniclot". ВСЕГДА соло, не комбинируется. |
| `collection_footer` | Три капс-блока `BRAND_LABEL \| TITLE \| CHARACTER`, чередование цвета по словам внутри блока (первое `dominant`, второе `accent`); мелкий кегль (`0.028 fig_w`). Для `mood=pop_trash` — шрифт `gothic` вместо `caps`, цитата (если не идёт отдельным `quote_bottom`) добавляется 4-й строкой подвала. |

4. **`mood`** (поле art_director) определяет палитровые роли/шрифтовую пару/набор
   режимов разом: `duotone_quote` -> `quote_bottom`+`kanji_on`+`collection_footer`
   (Permanent Marker+Anton+NotoSansJP); `fashion_editorial` -> `editorial` соло
   (Playfair Display+NotoSansJP); `pop_trash` -> `kanji_on(ghost)`+`collection_footer`
   (Cinzel+NotoSansJP).

**Выбор типографики в `batch_print.render_design`** (`text-style=auto`): если
`design["text_modes_v3"]` непустой массив — вызывается `typography_v3.compose_text_v3`
(режимы + палитра конкретной иллюстрации); иначе — старый путь v2
(`typography.compose_text` по `design["text_mode"]`). Явный `--text-style` (v1)
по-прежнему приоритетнее обоих.

**`BRAND_LABEL`** (`.env`, дефолт `"ANICLOT COLLECTION"`, читается в `config.py`) —
источник текста для `collection_footer`, не хардкод внутри `typography_v3.py`.

**`reprocess_typo.py` адаптирован под v3** (бесплатная проверка без повторной
оплаты): если в `design.json` есть непустой `text_modes_v3` — используется он как
есть; дампы v2 (есть `text_mode`, но нет `text_modes_v3`) — старый путь v2 как есть;
СОВСЕМ старые дампы (до седьмого захода, ни того, ни другого поля) — эвристика:
round-robin по каноничным комбинациям раздела 3.6 стайлгайда
(`quote_bottom+kanji_on+collection_footer` / `editorial` / `kanji_on+collection_footer`
(ghost) / `quote_bottom+collection_footer`), поля `quote`/`name_jp`/`mood` достраиваются
из существующих `slogan`/`kana` дампа (`_heuristic_v3_fields`). `--mode` CLI
по-прежнему форсирует один v2-режим на все дизайны (приоритетнее v3-путей).

## Разовый батч — запуск

```powershell
# 1. Заполнить .env (см. .env.example)
# 2. Установить зависимости
pip install -r requirements.txt

# 3. Файл тем: по одной теме на строку, строки с # — комментарии
python batch_print.py --file themes_test.txt --format diecut

# Явный override типографики (v1-стиль, приоритетнее text_mode) и ручной override хромакея
python batch_print.py --file themes_test.txt --format diecut --text-style kana --chroma blue
```

Результат в `out_batch/<timestamp>/`.

Флаги:
- `--file <путь>` (обязателен) — список тем, по одному дизайну на строку.
- `--format cutout|diecut` (дефолт `diecut`).
- `--text-style auto|none|anton|kana|comic|tag` (дефолт `auto`) — `auto` — типографика
  v2 (`typography.compose_text`), режим (`none`/`under`/`punch`/`kana_side`) решает
  арт-директор ПО КОМПОЗИЦИИ каждого дизайна (см. раздел «Типографика v2» выше). Явный
  стиль v1 (`none`/`anton`/`kana`/`comic`/`tag`, см. `typography.py`) ПРИОРИТЕТНЕЕ
  `text_mode`, если передан явно.
- `--chroma green|blue` — ручной override цвета хромакея для ВСЕХ дизайнов батча,
  обходит выбор арт-директора и код-предохранитель.
- `--retries N` (дефолт 2) — доп. попытки генерации при провале QC-гейта границ кадра.
- `--no-juice` — отключить пост-фильтр сочности цвета (Color x1.15 / Contrast x1.05).

## Ежедневный контур — тренды -> 500 принтов/день

### theme_scout.py — сбор и разворачивание тем

```powershell
python theme_scout.py                         # полный цикл: парсеры + план на PRINTS_PER_DAY
python theme_scout.py --target 40             # план на 40 заданий
python theme_scout.py --skip-collect          # без перезапуска парсеров trend-watch
```

Что делает:
1. Запускает `media_watch.py`, `anime_watch.py` и `pop_watch.py` из соседнего
   `trend-watch/` (путь — `.env: TREND_WATCH_DIR`, дефолт `../trend-watch`) как
   subprocess, с таймаутом — упавший/долгий/отсутствующий парсер не валит цикл, дальше
   работаем с тем, что уже лежит в CSV (может быть от предыдущего запуска или пусто).
   `pop_watch.py` — третий источник, pop-тренды (anilist/jikan/youtube/gtrends),
   появляется в `trend-watch/` независимо от этого проекта — его отсутствие
   ОБРАБАТЫВАЕТСЯ ТАК ЖЕ, как у двух других парсеров.
2. Читает `trend-watch/data/media_latest.csv`, `anime_latest.csv` (топ-8 и топ-6 по
   score) и `pop_latest.csv` (топ-8 по score, если файл есть), передаёт лемму +
   `example_text` Claude-«тематизатору». Строки `pop_latest.csv` маршрутизируются по
   колонке `sources`: `anilist`/`jikan` -> аниме-пул (там `lemma` = готовое название
   тайтла), `youtube`/`gtrends` -> общий пул (там `lemma` = заголовок видео/поисковый
   запрос).
3. Тематизатор разворачивает: аниме-тайтл/персонаж -> ростер главных персонажей
   (максимум 15-20 на один тайтл, не весь ростер) x форматы x 1-2 вариации образа;
   общая тема -> несколько принтопригодных концептов. Pop-темы передаются Claude
   ОТДЕЛЬНЫМ блоком промпта с явной пометкой «это готовые названия сущностей
   (тайтл/видео/запрос), превращай в задание напрямую, не ищи леммы внутри строки».
   **Строгий фильтр принтопригодности прямо в систем-промпте, одинаково для ВСЕХ
   блоков (медиа, аниме, pop)**: политика, война, катастрофы, трагедии, смерти
   реальных людей, криминал — отсекаются МОЛЧА (и СМИ-лента, и YouTube-тренды полны
   такими темами, на футболку нельзя).
4. Если тем после фильтра не хватает на дневной объём — добор из
   `evergreen_themes.txt` (104 вечнозелёные темы: топ-аниме персонажи, культовые
   машины, бытовые концепты), round-robin по кругу если нужно больше, чем строк.
5. Выход: `themes_daily_YYYY-MM-DD.txt` (плоский список тем) + `.json` (план с
   метаданными `theme/format/source`). `source` = `trend` (медиа/аниме-лента),
   `trend:pop:<anilist|jikan|youtube|gtrends>` (pop-лента, метка по конкретному
   источнику строки) или `evergreen`.

Ответы Claude парсятся как JSON с 1 ретраем, как в `art_director` — при двойном сбое
парсинга НЕ падает весь цикл, а честно печатает предупреждение и переходит на
чистый evergreen-добор (никогда не откатывается тихо на пустой/бредовый план).

### Источники трендов (три ленты `trend-watch/data/*_latest.csv`)

| Лента | Парсер | Что внутри `lemma` | Куда маршрутизируется |
|---|---|---|---|
| `media_latest.csv` | `media_watch.py` | лемма из СМИ/соцсетей | общий пул |
| `anime_latest.csv` | `anime_watch.py` | лемма аниме-комьюнити | аниме-пул |
| `pop_latest.csv` | `pop_watch.py` (соседний проект) | ГОТОВОЕ название: тайтл (anilist/jikan) или заголовок видео/запрос (youtube/gtrends) | по колонке `sources`: `anilist`/`jikan` -> аниме-пул, `youtube`/`gtrends` -> общий пул |

`pop_latest.csv` — контракт СТРОГО те же колонки, что у `media_latest.csv`
(`score,lemma,mentions,delta,is_new,sources_count,sources,example_text,example_url`),
но `lemma` там уже готовая тема (не сырое ключевое слово для дальнейшего разбора), а
`sources` — одно значение (`anilist`/`jikan`/`youtube`/`gtrends`), а не список СМИ.
Файл поддерживается соседним проектом `trend-watch/` — этот проект (`print-factory-nb`)
только читает CSV и запускает `pop_watch.py` subprocess'ом, сам его не пишет.

**YouTube-источник (`.env: YOUTUBE_API_KEY`)** — опционален, используется
`trend-watch/pop_watch.py`, не этим проектом напрямую. Бесплатный ключ Google Cloud
(включить YouTube Data API v3 в Google Cloud Console). Без ключа `pop_watch.py`
пропускает источник `youtube` (graceful degradation — остальные источники pop-ленты и
весь остальной контур работают как обычно).

### franchise_scout.py — «вскрытие франшизы» (измеренная популярность персонажей)

**Зачем**: Claude по своим знаниям не в курсе, кто из героев СВЕЖЕГО тайтла реально
выстрелил у фанатов ПРЯМО СЕЙЧАС — классический пример: в Re:Zero формально главная
героиня Эмилия, но у Рем сильно больше favourites на AniList, и фанаты покупают
принты именно с ней. `franchise_scout.py` подсовывает тематизатору ЧИСЛА из реальных
источников вместо догадок модели.

По названию тайтла (аниме/сериал/дорама/фильм) собирает сигналы из 5 источников
(каждый — независимая graceful degradation, падение одного не мешает остальным):

| Источник | Что даёт | Нужен ключ |
|---|---|---|
| AniList (GraphQL) | `favourites` на персонажа + роль MAIN/SUPPORTING | нет |
| Jikan (MyAnimeList) | `favorites` на персонажа | нет |
| TMDB | порядок каста (billing order) + popularity актёров — для неаниме | `TMDB_API_KEY` |
| YouTube | топ-эдиты по теме + просмотры (сигнал МОМЕНТОВ, не только персонажей) | `YOUTUBE_API_KEY` |
| Google Trends (trendspy) | rising related queries (что дописывают к тайтлу в поиске) | нет (неофициальная библиотека, капризная) |

Все 5 сигналов сводятся в ОДИН вызов Claude (`build_dossier`), который взвешивает
ЧИСЛА из входа (не свои знания) и возвращает JSON-досье: персонажи по убыванию
`score` (0-100) с `why` (какие числа привели к этому score) и `print_moment`
(конкретная сцена/арка/форма для принта). Парсинг — тот же приём, что в
`theme_scout._parse_scout` (ремонт обрезанного по `max_tokens` JSON + 1 ретрай +
дамп сырого ответа в `out_batch/scout_failures/` при двойном сбое — ЯВНАЯ ошибка,
не тихий откат).

**CLI (ручной инструмент владельца)**:
```powershell
python franchise_scout.py "Re:Zero"                    # kind=auto (пробует anilist/jikan+tmdb)
python franchise_scout.py "Re:Zero" --kind anime        # только anilist/jikan
python franchise_scout.py "Слово пацана" --kind tv      # только tmdb
```
Печатает ранжированных персонажей (score, why, print_moment) и топ-моменты
(название эдита/запроса + просмотры/эвиденс).

**Кэш** — `data/franchise_cache/<slug>_<YYYY-MM-DD>.json`: повторный запрос того же
тайтла в тот же день читает кэш и НЕ дёргает сеть (экономит квоту YouTube). Проверить
кэш просто: запустить CLI дважды подряд на один тайтл — вторая строка вывода будет
`франшиза '<title>': досье взято из кэша (<file>.json), сеть не дёргалась`, вместо
`собираю сигналы (kind=...)`.

**Экономика квоты YouTube**: `search.list` (поиск эдитов) стоит **100 юнитов**
из бесплатных 10000/сутки, `videos.list` (просмотры по id) — **1 юнит**. Функция
`_youtube_edits` делает СТРОГО один `search.list` на тайтл — итого один `build_dossier`
с реальным YouTube-сигналом тратит **~101 юнит**. Без `YOUTUBE_API_KEY` источник
просто пропускается (досье строится по остальным 4 сигналам).

**Интеграция в `theme_scout.py`** (`.env: FRANCHISE_DEEP_N`, дефолт 3): перед вызовом
тематизатора `build_daily_plan` выбирает `FRANCHISE_DEEP_N` самых высокоскоровых
аниме-тайтлов дня (из `anime_latest.csv` и pop:anilist/pop:jikan групп) и собирает
для них досье (`_collect_dossiers`) — падение ОДНОГО тайтла (сеть/Claude) печатает
предупреждение и не мешает ни другим тайтлам, ни всему циклу сбора трендов. Для
тайтлов с досье в промпт тематизатора добавляется блок «ДОСЬЕ ФРАНШИЗЫ (ИЗМЕРЕННАЯ
популярность — приоритет над твоими знаниями)» с ранжированными персонажами и
`print_moment`; `SYSTEM_SCOUT` прямо требует строить ростер ИЗ ДОСЬЕ по убыванию
score, а не по собственным знаниям Claude о том, кто в тайтле формально главный.
При недоступном `franchise_scout` (сеть упала на всех 5 источниках, Claude дважды не
собрал JSON) — тематизатор работает как раньше, без досье, просто с предупреждением
в консоли.

### daily_prints.py — дневной оркестратор

```powershell
python daily_prints.py --dry-run              # полный контур БЕЗ платных вызовов картинок
python daily_prints.py --dry-run --limit 5    # то же, но план обрезан до 5 заданий
python daily_prints.py --limit 2              # мини-боевой тест (2 платные генерации)
python daily_prints.py                        # полный день (PRINTS_PER_DAY, дефолт 500)
```

Что делает:
1. Собирает план дня (тот же код, что `theme_scout.py`).
2. Печатает смету (`цена_за_шт x N заданий`) по текущему `IMAGE_PROVIDER` — если смета
   превышает `.env: MAX_DAILY_COST_USD`, план обрезается ДО старта генерации.
3. Ведёт журнал `out_batch/daily_YYYY-MM-DD/queue.jsonl` (JSONL, одна строка на
   задание). При повторном запуске в тот же день — задания со статусом `done`
   пропускаются (не перегенерируем, не платим дважды); `failed`/`dry_run_skipped`
   обрабатываются заново.
4. Каждое задание проходит `batch_print.render_design()` (тот же пайплайн, что
   разовый батч) через `ThreadPoolExecutor` (`.env: WORKERS`, дефолт 4). QC-ретраи
   учитываются в поле `attempts` журнала.
5. `--dry-run` — весь контур (парсеры + тематизатор + план + смета + журнал) БЕЗ
   единого платного вызова генерации картинки — для проверки, что контур не падает.
6. `--limit N` — обрезать план для теста.

Результат в `out_batch/daily_YYYY-MM-DD/` (`plan.json`, `queue.jsonl`, и файлы
`NNNN_slug_raw/diecut/ongreen.png` + `_design.json` для каждого задания).

## Провайдер картинки (`.env: IMAGE_PROVIDER`)

`providers.py` — тонкая абстракция `generate_image(prompt) -> PIL.Image` с двумя
реализациями:

- **gemini** (ДЕФОЛТ — дешевле Pollinations в 3.5 раза, важно на объёме 500/день) —
  напрямую Google Gemini API, модель `.env: GEMINI_MODEL` (дефолт
  `gemini-3.1-flash-image` = Nano Banana 2, она же nano-banana), ключ
  `GEMINI_API_KEY`, обычный `requests.POST` без SDK. `generate_image`/
  `_generate_gemini` принимают опциональный `model` для точечного override
  (задел на `GEMINI_MODEL_PREMIUM` = `gemini-3-pro-image`, Nano Banana Pro).
- **pollinations** — модель `nanobanana` через `gen.pollinations.ai`, токен
  `POLLINATIONS_TOKEN`.

`providers.verify_text_in_image(image)` — ОТДЕЛЬНАЯ дешёвая точка входа (модель
`gemini-2.5-flash`, текстовая, НЕ image) для OCR-контроля спеллинга встроенного
текста (`TEXT_RENDER=image`, см. раздел «Текст в генерации» выше) —
`batch_print._verify_text` её вызывает после каждой генерации.

Переключение — одна строка в `.env`:
```
IMAGE_PROVIDER=gemini   # или pollinations
```

## Известные грабли (учтены в коде)

1. nano-banana честно рисует ровный хромакей-фон только по ЯВНОЙ просьбе с RGB-числом
   в промпте (`art_director._chroma_bg`) — просто "green background" даёт блёклый тон.
2. Код-предохранитель: если в идее/описании персонажа встречается green/зелён
   (регистронезависимо) — хромакей принудительно `blue`, даже если Claude выбрал green
   (`art_director._parse`). `ongreen`-версия всё равно кладётся кодом на RGB(0,177,64)
   независимо от того, какой хромакей был при генерации.
3. **(УСТАРЕЛО для NB>=2, см. раздел «Текст в генерации» выше)** Изначально считалось,
   что текст на принте можно наносить ТОЛЬКО кодом (`typography.py`) — ранние
   поколения nano-banana (`gemini-2.5-flash-image`) рисовали буквы с ошибками
   спеллинга или не рисовали вовсе, поэтому промпт картинки безусловно запрещал
   любые буквы/леттеринг/вотермарки. С `gemini-3.1-flash-image` (Nano Banana 2,
   десятый заход, `.env: TEXT_RENDER=image`, дефолт) это правило БОЛЬШЕ НЕ ВЕРНО —
   модель рисует встроенный текст без ошибок и гармоничнее кодовой типографики, OCR-
   контроль (`batch_print._verify_text`) подтверждает спеллинг после каждой
   генерации. Кодовый путь (`typography.py`/`typography_v3.py`, `TEXT_RENDER=code`)
   ОСТАЁТСЯ рабочим фолбэком — автоматически включается при провале OCR-контроля за
   все попытки, а также остаётся ручным режимом переключения через `.env`.
4. Слоган = каноничная коронная фраза персонажа (для не-персонажных тем — уместная
   короткая фраза), плюс `kana` — имя катаканой для стиля `kana`.
5. QC-гейт границ кадра (`_border_chroma_coverage` в `batch_print.py`): рамка 1% кадра
   должна быть >=90% хромакеем, иначе до 2 ретраев генерации, в конце берётся лучшая
   попытка.
6. В систем-промпт арт-директора — строгий запрет вторых существ/мехов/спутников/
   трансформаций за спиной персонажа.
7. Ответ Claude парсится как JSON; при сбое — 1 ретрай запроса, затем явная ошибка и
   пропуск дизайна (никогда не откатывается тихо на сырую тему). То же правило — у
   `theme_scout.py`-тематизатора (при двойном сбое переходит на evergreen-добор).
8. Windows-консоль (cp1251): `sys.stdout.reconfigure(errors="replace")` в начале всех
   CLI-скриптов, вывод без эмодзи.
9. Claude-тематизатор при попытке выдать ВЕСЬ дневной объём (напр. 500) ОДНИМ JSON-
   ответом обрезается по `max_tokens` на середине массива — `theme_scout.py` просит
   Claude за один вызов развернуть тренды в разумное число заданий (потолок
   `_MAX_TASKS_PER_CALL=60`, ростер персонажа не более 15-20 на тайтл), остальное
   добирается из evergreen. Если тематизатор всё равно не выдал валидный JSON дважды
   подряд — план полностью строится из evergreen (не падает).
10. `evergreen_themes.txt` пуст политикой/войной по построению (только персонажи/
    машины/бытовые концепты) — фильтр в systemprompt тематизатора нужен именно для
    ленты СМИ-трендов, которая реально полна такими темами.
11. `pop_latest.csv` (третий источник трендов, пишет соседний `pop_watch.py`) — тот же
    фильтр принтопригодности применяется и к нему: YouTube-заголовки (`sources=youtube`)
    так же часто содержат политику/трагедии/криминал, как СМИ-лента, поблажек нет.
12. Третий парсер (`pop_watch.py`) может отсутствовать или ещё не появиться в
    `trend-watch/` (создаётся параллельно другим разработчиком) — `theme_scout.py`
    обрабатывает это идентично отсутствию `media_watch.py`/`anime_watch.py`:
    предупреждение в лог, план строится без pop-блока, цикл не падает.
13. `franchise_scout.py` (глубокое досье персонажей для `FRANCHISE_DEEP_N`
    самых горячих аниме-тайтлов дня) — падение ЛЮБОГО из 5 сигнальных источников
    ИЛИ падение синтеза Claude для ОДНОГО тайтла не мешает ни остальным тайтлам,
    ни всему циклу `theme_scout.py`: предупреждение в лог, тематизатор работает
    по этому тайтлу как раньше (без досье, по своим знаниям).
14. `trendspy.related_queries` возвращает `pandas.DataFrame` (не список/dict) —
    прямая проверка `if rising:`/`rising or []` на DataFrame бросает
    `ValueError: truth value of a DataFrame is ambiguous`; в `franchise_scout.
    _gtrends_related` используется явный `isinstance`+`.to_dict("records")`
    вместо булевой проверки объекта.
15. Рисование ПО РЕФЕРЕНСУ (`character_ref.py`) поддержано ТОЛЬКО провайдером
    `gemini` — `pollinations` его игнорирует (с явным предупреждением), так как
    формат вызова `gen.pollinations.ai` для модели `nanobanana` не документирован
    как принимающий image-input в этом заходе.
16. Кэш референсов (`data/char_refs/<slug>.jpg`) переживает битые файлы — при ошибке
    чтения (`PIL.UnidentifiedImageError` и т.п.) `character_ref.py` печатает
    предупреждение и перезапрашивает сеть, не падает.
17. Канон-оружие/атрибут (`signature_props`) — арт-директор описывает его С ТОЙ ЖЕ
    строгостью, что и внешность персонажа (форма/цвет/отделка), а НЕ обобщает до
    "a sword"; вставляется явным предложением в промпт (`art_director.build_prompt`)
    и в `_REFERENCE_PREFIX` при рисовании по референсу.
18. Типографика v2 (`compose_text`) позиционируется от bounding box'а АЛЬФЫ ФИГУРЫ
    (реальные края конкретной вырезки), не от процентов холста — на непрямоугольных
    силуэтах (плащи, широкие низы) второстепенный текст (напр. sub-слоган в
    `kana_side`), центрированный на границе фигуры, может уйти ПОД силуэт и стать
    нечитаемым; лечится ростом текста НАРУЖУ от края фигуры/колонки, не центрированием
    на ней (см. `_compose_kana_side`).
19. `PIL.ImageDraw.text` для ОДИНОЧНОГО символа `" "` возвращает через `textbbox`
    advance width (не нулевой bbox), НО измерение через `draw.textbbox` ЦЕЛОЙ СТРОКИ
    с trailing space (`"CURSED "`) ОБРЕЗАЕТ этот пробел из bbox — курсор для
    позиционирования следующего разноцветного слова "съедал" пробел между словами
    (`typography_v3._measure_piece` меряет ПОСИМВОЛЬНОЙ суммой, НЕ через
    `draw.textlength()` целой строки — на курсивных/маркерных шрифтах textlength
    строки учитывает кернинг и даёт МЕНЬШУЮ ширину, чем реальная посимвольная
    отрисовка `typography._draw_spaced`, которая кернинг не учитывает: слова
    накладывались друг на друга при использовании textlength для курсора).
20. `local_luminance` (`palette.py`) — область ПОД фигурой в diecut обычно ПРОЗРАЧНА
    (alpha=0); `PIL.Image.convert("RGB")` даёт `(0,0,0)` для полностью прозрачных
    пикселей — если считать яркость по ним "как есть", область читается как чёрная,
    хотя реальный фон после печати (футболка) обычно светлый. `local_luminance`
    явно исключает пиксели с `alpha<=20` из расчёта и возвращает нейтральный светлый
    дефолт (`235.0`), если ВСЕ пиксели области прозрачны.
21. Advance width пробела у капс/готических шрифтов подвала (`Anton`/`Cinzel`) на
    мелком кегле подвала (`0.028 fig_w`) визуально сливается с letter-spacing самих
    букв — word-gap между словами внутри блока и вокруг разделителя `" | "`
    расширен до двойного пробела (`typography_v3` — `word_gap`/`sep` константы) для
    визуальной читаемости на печати, при том же логическом контракте позиционирования.
22. Google Fonts-репозиторий для Cinzel/Playfair Display содержит ТОЛЬКО variable
    font (`ofl/cinzel/Cinzel[wght].ttf`, `ofl/playfairdisplay/PlayfairDisplay[wght].ttf`)
    — статических Black-файлов, упомянутых в стайлгайде по имени, в текущем срезе
    репозитория НЕТ (404 при прямом скачивании); вес 900 фиксируется кодом через
    `set_variation_by_axes([900])`, тем же приёмом, что уже был для `NotoSansJP[wght].ttf`.

  `comfyui-print-server/`. `compose_text`/`TEXT_MODES` (типографика v2) — новый код
  этого (седьмого) захода, не перенесён ниоткуда — переиспользует существующие

  `_draw_spaced`, `_colors`), но с новой геометрией (bbox альфы фигуры, а не проценты
  холста).

  `juice()`, `drop_small_islands()` перенесены из `comfyui-print-server/batch_print.py`
  (эти функции не завязаны на ComfyUI — годятся как есть, только адаптированы под

  типографика v2 расширяет холст сама внутри `compose_text`.
- `art_director.py` — требования к идее (пол персонажа явно, 2-3 канон-приметы, полный
  силуэт с полями, слоган+kana, запрет вторых существ, выбор chroma) взяты из
  `comfyui-print-server/art_director.py::SYSTEM_DIECUT`; сами промпты картинки
  переписаны с нуля кинематографичной прозой под nano-banana (не danbooru-теги).
  Поля `character_en`/`title_en` — новые (шестой заход), нужны только
  `character_ref.py`. Поля `signature_props`/`text_mode` — новые (седьмой заход).
  Поля `text_modes_v3`/`quote`/`name_jp`/`mood` — новые (восьмой+ заход, типографика
  v3, схема из `docs/PRINT_STYLE_GUIDE.md` раздел 4.1). Терминология оружия ужесточена
  (2а в `_COMMON_RULES`) — точный японский/каноничный термин (katana/nodachi/
  zanpakuto/naginata и т.п.), не обобщённый "sword"/"blade"; для Кенпачи Зараки
  эталонный термин — изношенная КАТАНА-ДЗАНПАКТО (zanpakuto) с зазубренной кромкой,
  не тесак и не обобщённый "nodachi". Поле `type_spec` — новое (десятый заход, см.
  раздел «Текст в генерации»), `_COMMON_RULES` разбит на `_COMMON_RULES_BASE` +
  условный пункт (6) по `config.TEXT_RENDER` (`_common_rules()` — функция, не
  константа, читает `TEXT_RENDER` на момент каждого вызова, важно для тестов);

  `system_diecut()`. `build_prompt` — эталонный A/B-промпт оркестратора
  (`out_batch/ab_models/ab_models_text.py`) послужил образцом формата text-блока
  (exact-spelling построчно + запрет прочего текста).
- `typography_v3.py` — новый файл (восьмой+ заход), не перенесён ниоткуда.

  `_draw_spaced`, `_hard_shadow_text`, `_alpha_bbox`, `_crop_to_content`) напрямую —

- `generator.py` (market-content-bot) — источник паттерна вызова Pollinations

  эталонным A/B-скриптом оркестратора (`out_batch/ab_models/ab_models_text.py`).
  `drop_small_islands(protect_text_islands=...)` — точечное расширение существующей
  функции (перенесённой из `comfyui-print-server/batch_print.py` в приёмке MVP),
  сама логика защиты островов новая.
- `theme_scout.py` — источник данных (CSV-формат, парсеры) — `trend-watch/media_watch.py`
  и `anime_watch.py` (запускаются subprocess'ом, сам `trend-watch/` не изменён).
- `franchise_scout.py` — эндпоинты и паттерны обработки ошибок (429/Retry-After,
  таймауты, graceful degradation на []) сверены с уже рабочими провайдерами
  `trend-watch/providers/anilist.py`, `jikan.py`, `tmdb.py`, `google_trends.py`,
  `youtube_trending.py` (те же URL/параметры API, `trend-watch/` не изменён —
  `franchise_scout.py` не импортирует эти файлы, а независимо повторяет те же
  проверенные вызовы под свои данные — GraphQL-запрос AniList другой, нужны
  персонажи и favourites, а не media/trending).

## Известные ограничения / блокеры

- **Генерация картинок заблокирована на момент сдачи.** Общий токен
  `POLLINATIONS_TOKEN` (тот же, что во всех проектах студии) имеет нулевой баланс на
  платных image-моделях (`nanobanana`) — Pollinations отдаёт HTTP 402. Прямой ключ
  `GEMINI_API_KEY` для print-factory-nb не заведён — ключи, найденные в других
  проектах студии (`trend-prints/anime-news`, `ozon-print-bot`), НЕ были перенесены и
  протестированы без явного разрешения владельца (кросс-проектное переиспользование
  чужих ключей требует его личного подтверждения, не только координатора). Смотри
  `docs/PROJECT_STATE.md` за полной диагностикой и что нужно для разблокировки.
- Провайдер `gemini` написан по документации Google (`responseModalities: ["IMAGE"]`,
  `inlineData`/`inline_data` в ответе) — не проверен живым вызовом ни разу (нет
  рабочего ключа/баланса для тестирования).
- `theme_scout.py`-тематизатор вероятностно не укладывается в JSON за один вызов на
  очень насыщенных трендах (проверено: один прогон дал 59/60 распарсенных заданий,
  другой на том же коде — 0 из-за обрыва по `max_tokens`) — контур не падает
  (переходит на evergreen), но при желании максимизировать долю "из трендов" в плане
  дня можно снизить `_MAX_TASKS_PER_CALL` дальше 60.
- `franchise_scout.py::_gtrends_related` на реальной проверке (2026-07-08) стабильно
  ловил `API quota exceeded for related queries/topics` от `trendspy` (лимит именно на
  `related_queries`, отдельный от лимита `trending_now`, который использует
  `pop_watch.py`) — источник штатно деградирует в `[]` (см. try/except в функции),
  досье собирается по остальным 4 сигналам. Не блокер, просто источник результата в
  реальных досье редко непустой на данный момент.
